System Architecture

The system operates as a three-tier pipeline: Trigger (GitHub Actions), Orchestrator (FastAPI + LangGraph on Azure Container Apps), and Backing Services (Cosmos DB, Key Vault, Application Insights).

Agent Pipeline — The Inner Loop

The core of the system is a LangGraph StateGraph with five nodes connected by edges. A conditional edge after the Skeptic creates the critique-revise loop.

Critique-Revise Loop If the Skeptic rejects the proposed changes (e.g., a fix would break a foreign key or introduce dirty reads), the pipeline routes back to the Linter for another pass. This continues until the Skeptic approves or the configurable iteration limit (default: 3) is reached.

Key Architectural Patterns

State-Machine Orchestration (not Chain-of-Thought)

Each agent is a discrete node in a directed graph, not a link in a prompt chain. This means the flow is deterministic — the Skeptic's verdict controls routing via a conditional edge function, not an LLM's free-form reasoning. This prevents hallucinated routing and makes the system auditable.

Pure-JSON Agent Contract

Agents do not use LangChain's tool-binding mechanism. When tools are bound to an LLM, the model returns tool-call objects instead of content, which breaks structured parsing. Instead, each agent's system prompt mandates a specific JSON schema, and tools (like format_sql and schema_checker) are invoked in Python code after the LLM responds. A normalize_severity() function maps the unpredictable severity strings LLMs produce ("high", "moderate", "critical") to a strict error/warning/info enum.

Schema-Aware Analysis

The Performance Critic and Skeptic agents receive a human-readable summary of the database schema (tables, columns, indexes, foreign keys) injected directly into their prompts. This is derived from a Schema.json file that can be overridden per request, enabling the system to analyze SQL against any target database.

Partial State Updates

Each node function returns only the state keys it modifies, never the full ReviewState. This follows LangGraph's reducer pattern and prevents accidental overwrites between agents.

Project Structure

The codebase is organized by responsibility:

  • src/agents/ — Four agent modules + markdown system prompts
  • src/graph/ — State definition, node functions, edge logic, graph builder
  • src/tools/ — Schema checker, SQL formatter, changelog writer
  • src/models/ — Pydantic models for API request/response and findings
  • src/services/ — Cosmos DB and Key Vault clients
  • src/telemetry/ — OpenTelemetry + Application Insights initialization
  • tests/ — 12 tests: unit tests per agent, graph integration, API endpoint tests
  • infra/ — Bicep template, parameters, deploy script
