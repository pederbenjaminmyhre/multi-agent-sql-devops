Lessons Learned

This document captures the key technical lessons learned during the design, implementation, and deployment of the Multi-Agent SQL Review Pipeline. These insights are intended to inform future projects involving agentic AI, cloud-native deployment, and CI/CD integration.

LLM Integration

LLM Behavior Do not bind tools to the LLM when you need structured JSON output. When LangChain tools are bound to an LLM via llm.bind_tools(), the model returns tool-call objects instead of text content. This made the .content field empty, causing every agent to report "unparseable output." The fix was to remove tool binding entirely and instead call tools in Python code after the LLM returns its JSON response. System prompts were updated to mandate raw JSON output with no markdown fences.
LLM Behavior LLMs do not respect enum constraints — normalize aggressively. Despite explicit instructions to use "error", "warning", or "info", the LLM returned severity values like "high", "moderate", "critical", and "informational." This crashed Pydantic validation. The solution was a normalize_severity() function that maps any string the LLM produces to the three-value enum, with a fallback to "warning" for unknown values. Similarly, line_ref was returned as an integer instead of a string — type coercion at the parsing boundary was essential.
LLM Behavior LLMs wrap JSON in markdown fences even when told not to. Even with "no markdown fences" in the system prompt, models frequently return ```json ... ``` wrappers. Every agent parser must strip these fences before attempting json.loads(). This is defensive code that should be standard in any LLM-to-JSON pipeline.

Architecture & Design

Architecture State-machine orchestration is the right abstraction for multi-agent systems. LangGraph's StateGraph with conditional edges made the critique-revise loop trivial to implement and easy to reason about. Each node is a pure function that takes state and returns partial updates. The Skeptic's verdict deterministically controls routing — no prompt engineering required for flow control.
Architecture Inject context into prompts rather than using tool calls for read-only data. The Performance and Skeptic agents need schema information (indexes, foreign keys). Rather than having the LLM call schema-checker tools (adding latency and non-determinism), the schema summary is built in Python and injected directly into the prompt. This is faster, more reliable, and cheaper (fewer round-trips).
Architecture Partial state updates prevent subtle bugs in multi-agent pipelines. Each node function returns only the keys it modifies. Early in development, returning the full state from one node would accidentally overwrite fields set by a previous node. LangGraph's reducer pattern — where returned keys are merged into state — enforces clean boundaries.

Testing

Testing Mock the LLM at the response level, not the framework level. The FakeLLM fixture returns a pre-set JSON string from its invoke() method. This tests the full parsing pipeline (JSON extraction, severity normalization, type coercion) without any LLM calls. Mocking at the LangChain framework level would have hidden the exact bugs that surfaced during live testing.
Testing Live LLM tests are essential — mocks cannot catch output-format drift. All 12 unit tests passed with mocks, but the first live call failed because the LLM returned tool-call objects instead of JSON. A separate live integration test (even manual) is necessary to validate the real LLM contract.

Deployment & Operations

Operations Azure CLI has persistent TLS issues on Windows that require workarounds. The Azure CLI's internal Python HTTP client encountered ConnectionResetError (10054) on every long-running ARM operation (Bicep deployments, container updates, ACR builds). Short operations (group show, resource list) worked fine. The workaround was using curl or Python's urllib with ssl.CERT_NONE for large payloads, and docker push instead of az acr build for image uploads. Setting AZURE_CLI_DISABLE_CONNECTION_VERIFICATION=1 helped for login but not for deployments.
Operations GitHub Actions workflows need explicit permissions to post PR comments. The default GITHUB_TOKEN in Actions has read-only permissions. The workflow failed with "Resource not accessible by integration" (403) when trying to post a comment. Adding a permissions block with pull-requests: write and issues: write resolved this.
Operations Build the Docker image before deploying the Container App. The Bicep template creates the Container App referencing an image that doesn't exist yet in ACR. This causes the initial deployment to partially fail. The image must be pushed first, then the Container App updated. A two-phase deploy (infra first, image second) is the reliable pattern.
Operations Scale-to-zero is viable for portfolio projects with no cold-start penalty that matters. Azure Container Apps with min replicas set to 0 incurs zero compute cost when idle. Cold starts add 2-5 seconds on the first request, which is acceptable for a review pipeline where the LLM calls themselves take 10-30 seconds.
