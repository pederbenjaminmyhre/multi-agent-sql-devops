Multi-Agent "Inner Loop" DevOps for SQL Server

Autonomous Review-and-Fix Pipeline — Project Requirements Document

Project ID: Portfolio – Project 2
Platform: Microsoft Azure
Orchestration: LangGraph (Python) / Semantic Kernel (.NET 9)
Trigger: GitHub Actions / Azure DevOps

1. Executive Summary

This project delivers an autonomous, multi-agent code-review pipeline that prevents poor-performing or insecure T-SQL from ever reaching a production-ready branch. By embedding specialized AI agents directly inside the CI/CD inner loop, every SQL change is linted, performance-analyzed, documented, and validated before a human reviewer ever sees it.

Primary Objective Create a fully autonomous "Review-and-Fix" loop that enforces T-SQL quality, performance, and documentation standards at the pull-request level—deployed to Azure as a live, portfolio-grade demonstration.

The system is designed not only as a functional DevOps tool but as a professional portfolio centerpiece demonstrating mastery of agentic AI orchestration, cloud-native deployment, and database reliability engineering.

2.2 Multi-Agent Roster

AgentRoleResponsibilities
Orchestrator (Supervisor) Router & Coordinator Routes SQL scripts through the specialist pipeline; manages state transitions and the critique-revise loop
Agent A — The Linter Static Analysis Scans T-SQL for anti-patterns: SELECT *, missing NOLOCK hints, implicit conversions, deprecated syntax
Agent B — The Performance Critic Performance Analysis Evaluates code against a schema dictionary (Schema.json); identifies missing indexes, parameter-sniffing risks, and inefficient JOINs
Agent C — The Doc-Genie Documentation Auto-generates Markdown documentation and SQL Extended Properties for every new table or stored procedure
Agent D — The Skeptic / Validator Safety Validation Reviews suggestions from Agents A and B to ensure they do not introduce breaking changes; triggers re-linting when issues are found

2.3 Core Workflow — The Inner Loop

The following state machine defines the end-to-end review cycle. If the Skeptic finds an issue, the script is routed back to the Linter, forming a closed critique-and-revise loop.

Critique-Revise Loop If Agent D (the Skeptic) detects that a proposed fix introduces a breaking change or logical error, the script is automatically routed back to Agent A for re-analysis. This loop continues until the Skeptic approves or a configurable iteration limit is reached.

Agent Tooling

  • Schema Checker Tool: Validates SQL against a Schema.json file containing table definitions, column types, and existing indexes.
  • SQL Formatter Tool: Applies consistent formatting and syntax standardization to all output scripts.

Outputs

  • Cleaned, validated SQL script ready for merge
  • Auto-generated ChangeLog.md documenting every modification
  • Markdown documentation and Extended Properties for new database objects
  • JSON report posted as PR comments or used to block merges on critical findings

3.2 CI/CD Pipeline

The end-to-end deployment and review pipeline operates as follows:

  1. Trigger: A developer pushes a .sql file to a feature branch on GitHub.
  2. GitHub Action fires: The workflow authenticates with Azure using a Service Principal, extracts the SQL content, and sends it along with schema context to the Orchestrator API.
  3. Agent loop executes inside Azure Container Apps (Linter → Performance → Documentation → Skeptic, with critique-revise looping as needed).
  4. JSON report returned: The orchestrator sends a structured report back to the GitHub Action.
  5. PR integration: The Action posts automated review comments on the pull request and optionally blocks the merge if critical issues are found.

GitHub Actions Workflow Artifacts

  • Custom YAML workflow triggered on pull_request events
  • Service Principal credentials stored as GitHub Secrets
  • Authenticated calls to the Azure-hosted API endpoint

3.3 Cost Management

Scale-to-Zero Architecture Azure Container Apps on the Consumption tier and Cosmos DB in Serverless mode ensure that the system incurs minimal cost when idle. The agents only consume compute during active reviews, making this suitable for a portfolio project with intermittent traffic.

3.4 Security & Secrets Management

  • Azure Key Vault stores all sensitive values: database connection strings, LLM API keys, and service credentials.
  • Service Principal with least-privilege RBAC grants GitHub Actions scoped access to Azure resources.
  • Managed Identity on Container Apps eliminates the need for credentials in application code.
  • No secrets are stored in source control, environment variables, or Docker images.

3.5 Observability & Tracing

Enterprise stakeholders are wary of "black box" AI. This system provides full transparency into agent decision-making:

  • Azure Application Insights captures distributed traces across all agent invocations.
  • Each agent's reasoning chain, input, and output are logged as custom telemetry events.
  • Azure Monitor dashboards surface key metrics: review latency, loop iterations, and issue severity distribution.
  • All traces are queryable in the Azure Portal, enabling post-hoc analysis of any review decision.
